## 补充输入：两项关键前提修正

> 会议完成后，**评审人**提出了两项对全部决策有根本性影响的补充输入。以下是这两项输入的内容，以及基于它们对所有决策的重新评估。**最终决策以此处修订后的版本为准。**

### 输入 1：`###` 不应当是核心架构假设

> *"站在使用者的角度，sql 文件用 ### 来分隔似乎是一种累赘，之前的讨论包含了使用 ### 进行分隔这个假设。"*

**影响**：
- 如果 `###` 令人感到累赘，说明它不应该成为补全管道的核心设计要素。
- 补全管道不应假设 Lua 预先把文件切成 `###` 块再交给 Rust。Rust 应当具备处理完整文件内容的能力。
- `###` 应降级为**可选的执行层视觉分割符**，而不是补全层面的概念。
- 这意味着：原有的 "Lua 处理文件结构 / Rust 处理 SQL" 边界需要重新审视。

### 输入 2：无真实用户，破坏性变更没有代价

> *"当前不太需要考虑重构的破坏性，因为还没有用户使用，只是开发阶段，最重要的是确保后面工作的稳健性（太脆弱的设计会导致很多无用功）"*

**影响**：
- 所有基于"不要破坏用户"的论据都失效了。Agent A 的大部分立场需要重新评估。
- 我们可以选择**最干净的设计**，而不是"最小破坏的设计"。
- "太脆弱的设计会导致很多无用功"——这意味着 Agent C 的"折中方案"如果本质上是拖延问题，也需要修正。
- Agent B 的"干净设计优先"路线在当前阶段是更合理的。

### 重新评估的方法

以下重新审视全部 10 个决策，针对每项决策回答三个问题：
1. 旧决策是否依赖于 `###` 作为架构假设？
2. 旧决策是否依赖于"不破坏用户"的理由？
3. 在无 `###` + 无用户约束下，最优选择是什么？

## 议程 2：文件语法契约（15 分钟）

### 2.1 文件结构与扩展名统一

**讨论**：
- **Agent A**：`.sql`/`.mysql`/`.sqlite` 统一使用相同块规则是正确的。方言影响只在 SQL 上下文检测层，不影响文件结构。
- **Agent B**：当前设计合理，不需要额外统一。所有扩展名走同一套文件解析逻辑。
- **Agent C**：保持现状。如果未来需要新的文件格式扩展（如 `.pgsql`），按相同模式添加即可。

**决策**：✅ **文件结构保持当前定义。`.sql`/`.mysql`/`.sqlite` 共享相同块规则。扩展名仅影响方言选择。**

### 2.2 块级 `@connection` 是否重置 `@database`？

**讨论**：
- **Agent A**：块级覆盖应遵循"最后一次赋值"的语义——如果块内有 `@connection` 但没有 `@database`，`@database` 应继承文件头值。
- **Agent B**：块级覆盖应重置两者。原因是"选择一个新连接通常意味着要在一个新数据库上工作"。
- **Agent C**：应采用最小意外原则（Principle of Least Surprise）。大多数用户预期的行为是：块级 `@connection` 只改连接，不改数据库。如果用户想改数据库，他们会显式写 `@database`。

**决策**：✅ **块级 `@connection` 不重置 `@database`。Lua 实现应继承文件头的 `@database` 值，除非块内显式覆盖。**

## 议程 3：JSON 上下文契约（10 分钟）

### 3.1 Q4.1 & Q4.2：`version` 字段

见下方 D4 决策。

### 3.2 Q4.3：`statement`/`block` 字段

**讨论**：
- **Agent A**：在 Lua 中计算两者。`###` 是 Poste 文件格式信息，Lua 更靠近文件结构。"statement"可以通过调用 `poste context stmt` 获得。
- **Agent B**：如果未来 Rust LSP 需要自己了解这些信息，现在在 JSON 中占位（返回 null）以便契约先行定义。
- **Agent C**：当前 JSON 中不添加这两个字段。Lua 中需要时自己计算。契约文档中可以定义它们，但实际 Rust 响应中不输出 null 占位字段（减少 JSON 体积）。

**决策**：✅ **`statement`/`block` 字段在文档中定义，但 Rust 输出中暂不包含。Lua 在需要时自己计算。输入 1 之后，`block` 字段不再是补全概念（`###` 不是补全假设），属于执行层信息。`statement` 字段由 Lua 层计算足够。**

## 议程 4：设计决策讨论（20 分钟）

### D1：空行边界

| Agent | 原始立场 | 理由 |
|-------|----------|------|
| **A** | P1 保留，P3 移除 | 不冒回归风险 |
| **B** | P1 立即移除 | 简化 `context.rs`，认为 Lua 的 `###` 已防泄漏 |
| **C** | P1 保留 + TODO 标记 | 折中 |

**原始辩论摘要**：

Agent B 强调："当前 2+ 空行边界是技术债务。既然 Lua 已经处理了 `###` 块，Rust 层就不应该有另一套边界逻辑。"

Agent A 反驳："这是一个已知行为的、可预测的特性，不是 bug。"

Agent C 调和："建议暂时保留，加 TODO 注释。"

**重新评估**：

**关键问题**：D1 的原始辩论中，Agent B 的"Lua 的 `###` 已防泄漏"论据——在输入 1（`###` 不是架构假设）下**完全失效**。

没有 `###` 作为安全网后，空行边界是补全上下文中**防止跨语句表泄漏的唯一机制**。它不是一个可以移除的"临时启发式"，而是一个**必备的功能特性**。

| 评估维度 | 分析 |
|----------|------|
| 依赖 `###`？ | **是**。Agent B 的"移除"论据完全依赖 `###` 作为另一层分隔 |
| 依赖"不破坏用户"？ | 否 |
| 无 `###` + 无用户下的最优选择 | **永久保留**。P3 的 ScopeResolver 是**追加**（精化语句内的作用域），不是替代 |

**修正决策**：✅ **P3 前临时保留，P3 后由 ScopeResolver 替代并移除。空行边界是 `###` 移除后、ScopeResolver 完成前的临时防护，不是永久功能。**

### D2：`in_string`/`in_comment` 返回行为

| Agent | 原始立场 | 理由 |
|-------|----------|------|
| **A** | 选项 2（修复 CLI 回退） | 最小化变更 |
| **B** | 选项 3（显式 ctx_type） | 最干净的设计 |
| **C** | 选项 2（修复 CLI 回退） | 投入产出比最高 |

**原始辩论摘要**：

Agent B 坚持："选项 2 只是把问题推给了 Lua。如果我们要定义契约，就应该定义完整的契约。"

Agent C 回应："选项 3 的工作量是选项 2 的 3-5 倍，但用户体验没有区别。"

Agent A 加码："选项 3 有隐含风险——如果未来有人依赖 `ctx_type: 'keyword'` 来触发某种补全，但字符串内我们也返回 `keyword`，就会产生混淆。"

**重新评估**：

| 评估维度 | 分析 |
|----------|------|
| 依赖 `###`？ | 否 |
| 依赖"不破坏用户"？ | 部分。Agent A 的"隐含风险"论据假设有用户依赖当前行为 |
| 无 `###` + 无用户下的最优选择 | **选项 3（显式 ctx_type）**。既然没有用户会被破坏，我们应该选择更干净的契约。选项 2 只是在延迟问题——把歧义从 Rust 推给了 Lua，未来如果添加真正的"字符串内补全"功能，还是要做选项 3 的变更 |

**修正决策**：✅ **采纳选项 3。在 Rust 的 `ContextType` 枚举中添加 `String` 和 `Comment` 变体。光标本在字符串/注释内时，Rust 返回对应的显式类型而不是 `None`。Lua 层检查这些类型以抑制补全项提示。契约更清晰、自描述。P1 中实现。**

### D4：`version` 字段粒度

| Agent | 立场 | 理由 |
|-------|------|------|

| Agent | 原始立场 | 理由 |
|-------|----------|------|
| **A** | 永不让 Rust 感知 | `###` 是 Poste 语法 |

| **C** | 从议程移除 | 非 P0-P4 问题 |

**重新评估**：

| 评估维度 | 分析 |
|----------|------|

| 依赖"不破坏用户"？ | 否 |

### D8：指令归属

| Agent | 立场 | 理由 |
|-------|------|------|
| **A** | Lua 完全拥有 | 指令是 Poste 文件语法 |
| **B** | 两者保留，Rust 作为安全网 | 双重检查是安全的 |

**辩论摘要**：

Agent A 支持："安全和一致性的权衡中，一致性更重要。用户看到不一致的行为会困惑。安全网可以保留但只记录日志（warn），不返回 `Connection`。"

**决策**：✅ **采纳 Agent C 方案。Lua 完全拥有指令解析。Rust 的 `try_directive()` 保留但降级为安全网：遇到 `@connection`/`@database` 标记时返回 `None` 并记录 warn 日志（如果日志可用）。P1 中实现此变更。**

### 后续决定：完全移除 `###` + 内联指令 + 空行边界临时化

会议结束后的实施阶段，决定：

1. **完全移除 `###`** 作为 Poste 文件格式的分隔符。`###` 不再属于 Poste SQL 文件语法。
2. **引入内联指令机制**：`-- @connection` / `-- @database` 不限于文件顶部，可在 SQL 语句之间任意位置出现，从该位置起切换后续语句的有效连接/数据库，持续到下一个同类型指令或 EOF。
3. **空行边界改为 P3 前临时保留**：2+ 空行边界是 `###` 移除后、ScopeResolver 完成前的临时防护，P3 ScopeResolver 完成后移除，ScopeResolver 通过语句结构分析替代空行启发式。

对上述变化总结的更新：

| 维度 | 变化前 | 变化后 |
|------|--------|--------|
| `###` 在文件格式中的角色 | 执行层视觉可选分隔符 | **已移除，不存在于文件格式中** |
| `###` 在补全管道中的角色 | 补全层透明处理 | **不存在** |
| 空行边界 | **永久保留** | **P3 前临时保留，P3 后移除** |
| 文件结构 | 文件头 + 请求块（`###` 分割） | **指令 + SQL 语句（`;` 分隔），指令可在顶部或内联** |
| 指令作用域 | 块级覆盖（`###` 界定范围） | **前向最近指令生效（内联粘性）** |
| 边界检测（P3 后） | `;` + 2+ 空行 | **仅 `;`，由 ScopeResolver 处理作用域** |
